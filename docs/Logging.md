# Logging Architecture & Configuration

The Go License API features a production-grade logging system designed for high availability, observability, and performance.

## Architecture

The logging system is built on two primary components:
1.  **Fiber Logger Middleware**: Captures incoming HTTP requests and outgoing responses.
2.  **Lumberjack**: A rolling logger that handles log file rotation, compression, and retention.

### Key Features
- **Daily Log Files**: Automatically generates log files named by date (e.g., `2026-04-02.log`) when file output is enabled.
- **Request ID Tracking**: Every log entry is tagged with a unique Request ID, enabling end-to-end tracing of individual requests.
- **Fail-Safe Output**: If file-based logging is enabled but the directory cannot be created, the system automatically falls back to `stdout` to prevent application panics.
- **Configurable Retention**: Control how many old log files to keep and how large they can grow.

## Configuration

Logging is configured through environment variables, with sensible defaults for local development.

### Environment Variables

| Variable | Description | Default |
| :--- | :--- | :--- |
| `LOG_ENABLED` | Enables/Disables request logging. | `true` |
| `LOG_OUTPUT` | Destination for logs (`stdout` or `file`). | `stdout` |
| `LOG_DIR` | Directory for log files (if `LOG_OUTPUT=file`). | `./logs` |
| `LOG_LEVEL` | Minimum log level to record (e.g., `info`, `error`). | `error` |

### Log Rotation Constants
For fine-tuned control over log rotation, the following constants are defined in `configs/logging.go`:
- `MaxLogSize`: Maximum size in megabytes before a file is rotated (default: `100MB`).
- `MaxLogBackups`: Maximum number of old log files to retain (default: `7`).
- `MaxLogAge`: Maximum number of days to retain old log files (default: `7 days`).
- `LogCompress`: Whether to compress rotated log files using gzip (default: `true`).

## Log Format

The standard log format is designed for both human readability and machine parsing:

```text
${time} [${requestid}] ${status} - ${method} ${path}
```

- **${time}**: Timestamp of the request.
- **${requestid}**: Unique ID generated by the `requestid` middleware.
- **${status}**: HTTP response status code.
- **${method}**: HTTP request method (GET, POST, etc.).
- **${path}**: The requested endpoint.

### Example Entry
```log
04:13:44 [c4mN6GL7F4Zyy8g1GMxOHYTdvy...D0] 200 - GET /health
```

## Implementation Details

The logging setup is encapsulated in the `configs` package:
1.  `configs/logging.go`: Contains the `LoggingConfig` struct and the `Setup()` method that initializes the Fiber middleware.
2.  `internal/app/app.go`: Calls `logCfg.Setup(app)` during application startup.

## Best Practices

- **Containerized Environments**: In Docker or Kubernetes, it is recommended to keep `LOG_OUTPUT=stdout` to allow the orchestration layer (e.g., ELK stack, Splunk) to handle log aggregation.
- **Production Persistence**: Set `LOG_OUTPUT=file` if you need to persist logs directly on the host machine or a mounted volume. Ensure the `LOG_DIR` has appropriate write permissions.
